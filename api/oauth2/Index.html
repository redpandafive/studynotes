OAuth 2

  • An open protocol to allow secure authorization in a simple and standard method from web, mobile, and desktop applications
    • Authorizes an application
    • Client applications can request an access token to gain access to an API
    • Token is passed to the API to gain access
  • The standard defines how to use endpoints for different types of client applications
    • Identity providers should implement this standard
  • By itself, it does not provide authentication
  • There are four players
    • Protected resource (HTTP API)
    • Client application (requesting application) that wants to access the protected resource
    • Resource owner (the user) who owns the protected resource
    • The authorization server
      • Handles authorization requests
      • Trusted by all parties involved

Criticisms and Misconceptions

  • OAuth is not authentication
    • Access tokens do not represent the user (or the client application)
      • Intended audience is the protected resource
      • Client applications should have no need to look inside the validation token or try to validate it
      • Client applications just request the token and then pass it on to the protected resource
      • Client cannot reliably verify the access token
    • The authorization server has no obligation to clients regarding acces stoken, format, or scheme
      • Do not try to extract information from the token to use within the client to process authentication
