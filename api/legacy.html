Legacy Solutions

Legacy Technologies

  • Applications used be restricted to an internal network or a single domain
    • For instance, a service side web application that uses internal APIs
    • One protocol used was WS-Security
      • Microsoft impelemntation was WCF
    • Most used SAML 2.0 = standard for exchanging authentication and authorization data between security domains
  • The landscape eventually evolved to using Web APIs or Rest applications (Service based applications)
    • No longer nessarily lived in the same domain and possibly were not under developer control
    • Client applications could use its own API and other APIs as well, such as Google Map
  • Client applications now often require public APIs

Legacy API Security

  • Credential Sharing
    • No way for the API to tell the user and application apart
    • The application is impersonating a user (using the username and password)
    • No way to restrict single users from having access, permissions
    • No real way to revocate credentials without impacting all applications using the credentials
    • Does not allow federation
  • Cookies
    • Unsuitable for API security due to Cross-Site Request Forgery attacks (CSRF or XSRF)
  • API Keys
    • User uses a given API key so connec to the API
    • No longer expose user credentials
    • Still struggle with application tha cannot keep a secret, such as a single page app that does not have backend storage
    • No standard for creating keys
    • Are long lived, which can be a problem if it the key is stolen and the theft is not noticed
      • Short key lifetimes can cause frustrations with consumers of the API

Token Based Security

  • Alternative to sending usename and passwords with each request
  • Tokens are passed instead from the client to the server to grant access to resources
    • Tokens are granted on succesful login
    • Tokens are vlidated on the server on each request
  • Home grown token services emerged
    • Typically started with a login page where you entered your username and password
    • This would return a JSON Web Token
    • This still requires sharing your username and password with the api
      • That may be ok for your own trusted application, but not a good idead for 3rd part applications
    • End up reinventing the wheel each time implemented, including the following concerns:
      • Token expiration
      • Token signing and validatin
      • Token format
      • Authentication and authorization
