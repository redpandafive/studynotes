Headers

  • Many data services can return multiple data formats. It is up to the client to include a header field to indicate the data format that should be used.

URL components

  • URLS are composed of several parts
    • scheme:[//host[:port]][/]path[?query][#fragment]
  • URLS are a mixture of case insensitive parts and potentially case sensitive parts
  • Scheme
    • Case insensitive
  • Host
    • Case insensitive
  • Path
    • Contains data
    • Case sensitive
  • Query
    • Contains non-hierarchical data
    • Case sensitive
  • Fragment
    • Prefixed by #
    • Used to link directly to a section of the page
    • Ignored by the server when processing a request, it is only used by the browser
    • Case sensitive

Basics

  • HTTP communication is an asynchronous communication pattern
    • Client sends a request
    • Servier sends a response
  • Evolution of asynchronous communictions in web development
    • Callbacks - function that is executed when the reponse returns from a server
    • Promises - a promise object represents a single value that will be returned in the future
    • Observables - see RxJS

REST Services

  • Representational State Transfer
  • Basic idea is that you will be transfering resources (data objects) and taking advantage of the HTTP protocol to indicate the action that should be taken with those resources.
  • Most common way to deliver data to a web application
  • Also called a Web API or HTTP API
  • Many data services can return multiple data formats. It is up to the client to include a header field to indicate the data format that should be used.
  • Use HTTP versbs to specify CRUD operations
    • Create
    • Read
    • Update
    • Delete
  • URL conventions address individual as well as collection of resources
  • Use HTTP response codes to indicate success/failure of server action

REST Services - Create Operations

  • Creating a resource on the data service (insert)
  • Uses HTTP verb POST
  • The URL should represent the collection of the resource
    • For example, http://localhost/api/books
  • Sends the new resource in the body of the request
  • If successful, returns
    • HTTP 201 Created status code
    • The new created resource should be sent in the boyd of the response

REST Services - Read Opearations

  • Reads resources from a data service
  • Uses the HTTP verb GET
  • The URL should indicate what you are retrieving
    • The entire collection - http://localhost/api/books
    • An individual resource using its id - http://localhost/api/books/5
  • If successful, returns HTTP 200 OK status code

REST Services - Update Opearations

  • Updates a resource from a data service
  • Uses the HTTP verb PUT
  • The URL should indicate the specific resouce you are updating (includes id) - http://localhost/api/books/5
  • The body of the request should contain the resource itself
  • If successful, returns HTTP 204 No Content status code

REST Services - Delete Opearations

  • Deletes a resource from a data service
  • Uses the HTTP verb DELETE
  • The URL should indicate the specific resouce you are deleting (includes id) - http://localhost/api/books/5
  • If successful, returns HTTP 204 No Content status code
