Commits

  • Atomic Commits
    • Only do one change - it can be accross multiple files, but it is changing one thing
    • Keep changes small
    • Work on one thing at a time
    • Each commit should be able to get built
    • Should be reversable in case it breaks the build
  • Commit early, commit often
    • Commit locally all the time
    • Push when everything is working
  • Don't commit the following files
    • Generated files
      • Use .gitignore to exclude generated or compiled files
    • Dependencies
      • Use a package manager to manage dependencies and keep them out of the repository
    • Local configuration files
      • Allows users to have different settings
      • Files may have secrets, such as passwords or API keys
  • Don't commit broken code
    • This blocks other developers from working on their tasks
    • Destabilizes the code repository
    • Consider using the stash feature instead
  • Test changes before committing
  • Write useful commit messages
    • Should be straightforward
    • Should be meaningful
    • Use reference numbers if your team uses them
  • Refer to commits by their hash code
    • This uniquely identifies the commit
    • This allows you to use the git show command to see the contents
  • Have a good backup in place for the repository
    • Backup and version control are seperate concepts and should be used together
    • One is not a replacement for the other
