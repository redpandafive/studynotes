Stages of a File

  • Untracked
    • Files that are not part of the Git repository, but found in the project directory
    • Untracked files that are added to the project will be in the Staged state at first.
    • These will stay in the directory no matter which branch you are in
  • Tracked
    • Tracked files can be in one of three states
    • Committed
      • Data is stored in the repository
      • There have been no modifications since the last commit snapshot
      • When a committed file is changed, it is moved to the Modified state
      • These files belong to the branch where the git commit was run
    • Modified
      • The file has changed, but changed have not been recorded in the repository
      • Once you have finished modifying a file, you must move it to a Staged status for commit
    • Staged
      • A file that has been modified and you have identified that you want to move those files to the repository on the next commit
      • Once committed, files go back to a Committed status
      • If these have not been committed before, these files will stay in the staged state when switching branches

Three Main Parts of a Git Project

  • Working Directory
    • Holds a checkout of the project from .git directory
    • Files can be modified here
    • Changes that should be sent in the next commit must first be sent to the staging area - otherwise they will not be included.
  • Staging Area
    • Also known as Index
    • Used to build a up a change or set of changes that will sent in the next commit
    • Only changes in staging area are sent in a commit, not all changes in the working directory
  • .git Directory
    • Aka Repository
    • Origin of project data - this is what is pulled down from a remote server when copying a repository
    • Stores metadata, objects for projects

Commands

  • git show xxxxxx
    • Displayes contents of the object with identifier xxxxxx
    • For tags, it shows the tag message and the referenced objects
    • For trees, it shows the names
    • For plain blobs, it shows the plain contents

.gitignore File

  • File where you can tell Git to ignore certain files
    • These are files generated by code base, such as compiled code, log files, individual IDE setting files
    • Ignored files are not added to branchs, so they will not disappear and reappear when you switch branches
  • Good practice for a team to have a shared .gitignore file so everyone is on the same page
  • This should go in the project root
  • Syntax - uses uses regular expression-esque syntax to express patterns of files to ignore
    • * = one or more wildcard (except slash)
    • ? = single character wildcard (except slash)
    • ! = negator
    • / = directory seperator
      • These are evaluated relative to the directory where .gitignore resides
    • [a-zA-Z] = range of all alphabetic characters
    • # will prefix comments
    • ** will match any directory in the repository
      • **/bin will ignore any bin directory in the repository
    • *.zip to match any filename in the repository (this ignores any zip file)
  • Prebuilt .gitignore files can be found at https://github.com/github/gitignore

Exclude Files For Your System Only

  • The .git/info/exclude file will exclude files on your local repository only
  • Uses same syntax as the .gitignore file
