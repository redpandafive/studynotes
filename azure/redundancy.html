Storage Redundancy

Used to store massive amounts of unstructured data
  • Serving images or documents directly to a browser.
  • Storing files for distributed access.
  • Streaming video and audio.
  • Writing to log files.
  • Storing data for backup and restore, disaster recovery, and archiving.
  • Storing data for analysis by an on-premises or Azure-hosted service.

  • Azure stores multiple copies of data by default. Redundancy mainly addresses availability of data in the event of an incident
  • Azure Storage services are managed through a common Azure resource called a storage account
    • A shared pool of storage that can ube used to deploy storage resources, including
      • blobs
      • file shares
      • tables
      • queues
  • The Redundancy setting for a storage account is shared by all resources in that account
    • Seperate storage accounts can be used if different redundance requirements are needed
  • Data is an Azure Storage account is always replicated three times in the primary region
  • Options for how data is replicated in the primary region
    • Locally redundant storage (LRS)
    • Zone Redundant Storage (ZRS)

Single Region Redundancy

Locally redundant storage (LRS)

  • Lowest cost, but least durable option
  • Data is replicated three times within a single data center in the primary region
  • Protects against drive or server rack failures
  • May not protect against fire or flood in the data center
  • Good choice if
    • Data can be reconstructed if data loss occurs
    • Data is restricted on regions it can be replcated in

Zone Redundant Storage (ZRS)

  • Storage account is synchronously replicated across three Azure availability zones in the primary region
    • Each zone is a seperate physical location with independent power, cooling, and networking
  • Recommended for
    • High availability requirements
    • Restricting replication of data to a particular country or region to meet data governance requirements
  • May not protect against regional disasters
  • Does not support archive tier blob storage

Second Region Redundancy

  • Data in a storage account is copied to a secondary region that is distant from the primary region
  • You can select a primary region for a storage account, but not the paired secondary region
    • Secondary region is determined based on the primary region
    • It cannot be changed

Geo-Redundant storage (GRS)

  • Three copies within a single physical location in the primary region using LRS
  • Three copies in a single physical location in the secondary region using LRS
  • No read / write access unless there is a failover to the secondary region
    • Data will be unavailable until the failover process completes
    • Use read-access geo-redundant storage (RA-GRS) to have read ability

Geo-Zone-Redundant storage (GZRS)

  • Copies data across 3 availability zones in the primary region using ZRS
  • Copies data to a single location in the secondary region, where it is copied three times using LRS
  • No read / write access unless there is a failover to the secondary region
    • Data will be unavailable until the failover process completes
    • Select read-access geo-zone-redundant storage (RA-GZRS) to have read ability
