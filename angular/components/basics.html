Basics

  • Fundamental building blocks of Angular applications
  • Consists of
    • Class that contains application data and logic
    • An associated HTML template that defines a view to be displayed
    • An optional CSS file for styles used in the component
  • Define views
    • Sets of screen elements that Angular can choose among and modify according to your program logic and data
    • Allows construction of a view hierarchy, which allows you to define arbitrary complex areas of the screen that can be created, modified, and destroyed as a unit
  • Use services
    • Provide specific functionality not directly related to views
    • These are injected into components as dependencies
  • Every Angular application has at least one root component
    • Foundation component for an Angular project, so all applications require one and only one
    • Called app-root
    • When you bootstrap with the AppComponent class (in main.ts), Angular looks for a app-root in the index.html, instantiates an instance of AppComponent, and renders it inside the app-root tag
    • Connects a component hierarchy with the page DOM

Configuration

  • Uses a @Component decorator
    • This must be imported from @angular/core
    • Identifies the class immediate below as a component
  • Takes a metadata object with the following properties
    • selector – CSS selector that tells Angular to create an insert an instance of this component wherever it finds the corresponding tag in the template HTML
    • templateUrl - module-relative address of this component's HTML template
    • template – alternative to the templateUrl that gives the HTML template inline
      • Use backticks to surround it if it spans multiple lines
    • styles – takes an array of strings that contain CSS rule declarations
    • styleUrls – takes an array of CSS stylesheets that should be added
    • providers – an array of providers for services that the component requires

Best Practices

File Recommendations

  • Keep templates and styles in separate files (when over 3 lines long)
  • Limit code to 400 lines per file
  • Limit functions to 75 lines
  • Declare variables as cont when they will have constant values
  • Use lower camel case for properties and methods
  • Leave an empty line between imports and modules
  • Do not name interfaces starting with an uppercase I

Component Selectors

Component selectors should be
  • Dashed-case or kebab-case (toh-hero-button)
  • Include a prefix indicating the feature area or the application itself
  • Easier to identify and prevents naming collisions

Class Definitions

  • Use strongly typed objects for type safety
    • Avoid any types since you lose type safety
  • When using lifecycle hooks, implement lifecycle hook interfaces

Properties

  • Organize members as such
    • Public properties, alphabetized
    • Private properties, alphabetized
    • Public methods, alphabetized
    • Private methods, alphabetized

Component Logic

Keep logic minimal and only focused on that required for the view
  • Components should be small and focused
  • Delegate complex, duplicated logic to services
  • Put presentation logic in the component class, not the template

Component Communication

  • The correct way of component communication is to
    • Let parent components pass data to their children
    • Let children notify the parents when an action has been performed

Avoid Rewriting Single Elements as Components

Minimize writing custom components that replace or encapsulate single elements
  • Example of this would be replacing as single input element with a my-input component
  • Problems that arise
    • Consumer cannot customize the attributes of the native element unless the component API allows it
      • HTML element APIs are usually quite extensive, so a single element custom component limits what can be done with it
      • HTML element APIs are usually quite extensive, far more than can be economically replicated in a custom component
    • Native elements have built in accessibility attributes that are overwritten by implementing them as a component
    • Native elements are an API that consumers can use that they are already familiar with
      • Must relearn a different API on custom components
  • Better solution is to use directives to augment existing elements
    • Keeps the native elements exposed
    • Angular Material takes this approach

Assorted

  • Use ARIA attributes to describe what a component does
