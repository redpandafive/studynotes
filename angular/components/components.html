Basics

  • Fundamental building blocks of Angular applications
  • Consists of
    • Class that contains application data and logic
    • An associated HTML template that defines a view to be displayed
    • An optional CSS file for styles used in the component
  • Define views
    • Sets of screen elements that Angular can choose among and modify according to your program logic and data
    • Allows construction of a view hierarchy, which allows you to define arbitrary complex areas of the screen that can be created, modified, and destroyed as a unit
  • Use services
    • Provide specific functionality not directly related to views
    • These are injected into components as dependencies
  • Every Angular application has at least one root component
    • Foundation component for an Angular project, so all applications require one and only one
    • Called app-root
    • When you bootstrap with the AppComponent class (in main.ts), Angular looks for a app-root in the index.html, instantiates an instance of AppComponent, and renders it inside the app-root tag
    • Connects a component hierarchy with the page DOM

Configuration

  • Uses a @Component decorator
    • This must be imported from @angular/core
    • Identifies the class immediate below is as a component
  • Take a metadata object with the following properties
    • selector – CSS selector that tells Angular to create an insert an instance of this component wherever it finds the corresponding tag in the template HTML
    • templateUrl - module-relative address of this component's HTML template
    • template – alternative to the templateUrl that gives the HTML template inline
    • use backticks to surround it if it spans multiple lines
    • styles – takes an array of strings that contain CSS rule declarations
    • styleUrls – takes an array of CSS stylesheets that should be added
    • providers – an array of providers for services that the component requires

Input / Output Properties

  • When binding to a template to its own component class, you have no need of these.
  • These are used when binding to a property of a different component
    • TypeScript access modifiers can’t be used to shape a component's public binding API
  • Directive properties that must be declared as inputs or outputs

Input Properties

  • Settable properties; receive data values from parent components
  • Values flow into the property when it is data bound with a property binding
  • Declare using Input decorator in the child class and bind the property in the parent component template
  • Use ngOnChanges to detect when input property values change
    • This method receives a SimpleChange object
    • This will give you the old and new values of the inputs
  • Performance considerations
    • If you are passing a static value from a parent to a child, consider using the @Attribute decorator over @Input

Output Properties

  • Values flow out of the component as events bound with an event binding
  • This allows a child to send data to a parent component
  • Observable property = means you can subscribe to it
  • Annotated with @Output decorator
  • Events through outputs behave differently in Angular than they do in JS
    • Events do not bubble up to higher components like they do in JS
    • An event dispatched from a child component can only be heard by its parent component, not any higher
    • This was a design decision done to streamline concepts
    • A service bus design or store solution might be alternatives if this is needed (Can implement in plain JS, but this should be avoided)
  • Almost always returns an Angular EventEmitter object

In the child component, declare an EventEmitter property with the Output decorator

@Output() DeleteRequest= new EventEmitter<Hero>()

To dispatch the event in the child, raise it with the emit method call. Hero is the data you wish to pass back to the parent

this.DeleteRequest.emit(hero)

To configure the parent, add a method in the component to handle the data and bind that method to the child component template. The $event object contains the data dispatched by the event

<app-item-output (deleteRequest)="addItem($event)" ...>

Binding to an output property in a parent component. This calls a method onVoted on the parent component. The parameter for this method will be the payload sent by the child

<my-component (onvoted)="onVoted($event)"></my-component>

Import / Export Aliases

  • These are useful when you want to expose one name for a property, such name it something different internally
  • This can be done on both Inputs and Outputs
  • Style guide recommends this should be avoided when possible

Example of export alias

@Output('myClick') clicks = new EventEmitter<string>()

Property Setters

  • Intercept input property changes can be done with a setter –
  • This is useful if you need to do some sort of validation or standardization on inputs
  • Angular Fundamentals class uses this to create a derived value from an input property
    • Uses this to set the color of a voting icon to set or not set
    • The setting on an input property toggles the color of the icon
    • Eliminates the need for a conditional on the icon
  • Parent components can bind to output event properties on children and react to those events

    • Example of an input setter. This can then be used to reference an internal variable _name

    @Input() set name(name: string) {...} get name(name: string) { ... }

    ContentChild Expand

    • @ContentChild
      • Returns a reference to the first matching component or directive within the content of your component
    • @ContentChildren
      • Returns a reference to all matching components or directives within the content of your component

    Parent / Child Communication Through Local Variables

    • Allows a parent component to interact with child properties and invoke child methods
      • This cannot be done with data binding since it cannot read child properties or invoke child methods
    • Add a local variable on the child component
      • This gives you a reference to the child component and the ability to access any of its properties or methods from within the parent template
    • All wiring must be done entirely within the parent template
      • The parent component itself has no access to the child
      • This cannot be used if an instance of the parent component class must read or write child component values or must call child component methods
    • To implement
      • Add a template reference variable to the child component
      • The parent template can now reference the public methods and properties on the child

    Example

    <countdown-timer #timer>

    Parent/Child Communication Through Service

    • Parents and children can communicate via a shared service whose interface enables bi-directional communication within the family
    • Useful to define a Subject within this service
      • Have components subscribe to an Observable from this subject
      • Use the next() method to send a message to all subscribed components
      • Make sure to remove all subscriptions in the ngOnDestroy hook
