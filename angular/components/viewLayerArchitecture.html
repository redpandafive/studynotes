View Layer Architecture

  • Components can be separated by function
    • Keeps components small, simple and promotes reuse
  • Smart components
    • Aka application-level components, container components
    • Usually tied to an application since reuse is difficult
    • Has too many application-specific features, such as services injected into the constructor
    • Can be seen as a top-level component composed of one or more presentation components
      • Top level component of a route is usually a smart component
      • This component handles getting data from services
      • Data can be passed into a child presentation component for display
  • Presentation components
    • Aka Pure components, dumb components
    • Handles issues like formatting data into a table on the screen
    • Take data using inputs and emit events to
    • Try to make as application agnostic as possible
      • Avoid service injection that ties it to a specific application
      • Use inputs and outputs to get data into them
    • This is something that would be a good candidate for use in another application or in multiple places within the same application
  • One complication is event handling
    • Child components can only dispatch events to their parent component
    • Events do not bubble like they do in JS
    • A store-like service can be a solution if this is needed
  • These categories are guidelines or starting points
    • Smart components do not have to be top level component only
    • You may find yourself forced to make a presentation component into a smart component when need a service in it
    • It may not be worth trying to keep a component a presentation; you may sacrifice simple design for the sake of reusability
    • Design wins over reusability in this case
    • Start with a single top level smart component and split it into smaller presentation components as it gets bigger
    • With event handling, you could end up having to chain event listeners all the way down the component tree to stick with this paradigm
    • Probably better to make a leaf node a smart component in this case as it simplifies design
