Basics

  • Fundamental building blocks of Angular applications
  • Consists of
    • Class that contains application data and logic
    • An associated HTML template that defines a view to be displayed
    • An optional CSS file for styles used in the component
  • Define views
    • Sets of screen elements that Angular can choose among and modify according to your program logic and data
    • Allows construction of a view hierarchy, which allows you to define arbitrary complex areas of the screen that can be created, modified, and destroyed as a unit
  • Use services
    • Provide specific functionality not directly related to views
    • These are injected into components as dependencies
  • Every Angular application has at least one root component
    • Foundation component for an Angular project, so all applications require one and only one
    • Called app-root
    • When you bootstrap with the AppComponent class (in main.ts), Angular looks for a app-root in the index.html, instantiates an instance of AppComponent, and renders it inside the app-root tag
    • Connects a component hierarchy with the page DOM

Configuration

  • Uses a @Component decorator
    • This must be imported from @angular/core
    • Identifies the class immediate below is as a component
  • Take a metadata object with the following properties
    • selector – CSS selector that tells Angular to create an insert an instance of this component wherever it finds the corresponding tag in the template HTML
    • templateUrl - module-relative address of this component's HTML template
    • template – alternative to the templateUrl that gives the HTML template inline
    • use backticks to surround it if it spans multiple lines
    • styles – takes an array of strings that contain CSS rule declarations
    • styleUrls – takes an array of CSS stylesheets that should be added
    • providers – an array of providers for services that the component requires

Content Projection

  • Allows you to insert content into a child component from the parent component
  • Uses the ng-content directive as a placeholder in the child component
  • Does not create a real DOM element, so custom attributes applied to it are ignored
  • Single slot content projection
    • Component accepts content from a single source
    • Place a ng-content tag in the child component where external content should be placed
    • Put context between the opening and closing tags of the child component (in the parent component)
  • Multi slot content projection
    • Component accepts content from multiple sources
    • Each slot uses a selector to determine which content goes into that slot -this could be a
      • CSS selector (not recommended)
      • Attribute selector (recommended)
    • To implement
      • Place a ng-content tag in the child component for each slot
      • Tag each slot with a select attribute to indicate the selector
      • The square brackets make this an attribute selector
      • Add the selector as an attribute on the parent component
  • Conditional content projection
    • Will render content only when specific conditions are met
    • See documentation at https://angular.io/guide/content-projection#conditional-content-projection

Input / Output Properties

  • When binding to a template to its own component class, you have no need of these.
  • These are used when binding to a property of a different component
    • TypeScript access modifiers can’t be used to shape a component's public binding API
  • Directive properties that must be declared as inputs or outputs

Input Properties

  • Settable properties; receive data values from parent components
  • Values flow into the property when it is data bound with a property binding
  • Declare using Input decorator in the child class and bind the property in the parent component template
  • Use ngOnChanges to detect when input property values change
    • This method receives a SimpleChange object
    • This will give you the old and new values of the inputs
  • Performance considerations
    • If you are passing a static value from a parent to a child, consider using the @Attribute decorator over @Input

Output Properties

  • Values flow out of the component as events bound with an event binding
  • This allows a child to send data to a parent component
  • Observable property = means you can subscribe to it
  • Annotated with @Output decorator
  • Events through outputs behave differently in Angular than they do in JS
    • Events do not bubble up to higher components like they do in JS
    • An event dispatched from a child component can only be heard by its parent component, not any higher
    • This was a design decision done to streamline concepts
    • A service bus design or store solution might be alternatives if this is needed (Can implement in plain JS, but this should be avoided)
  • Almost always returns an Angular EventEmitter object

In the child component, declare an EventEmitter property with the Output decorator

@Output() DeleteRequest= new EventEmitter<Hero>()

To dispatch the event in the child, raise it with the emit method call. Hero is the data you wish to pass back to the parent

this.DeleteRequest.emit(hero)

To configure the parent, add a method in the component to handle the data and bind that method to the child component template. The $event object contains the data dispatched by the event

<app-item-output (deleteRequest)="addItem($event)" ...>

Binding to an output property in a parent component. This calls a method onVoted on the parent component. The parameter for this method will be the payload sent by the child

<my-component (onvoted)="onVoted($event)"></my-component>

Import / Export Aliases

  • These are useful when you want to expose one name for a property, such name it something different internally
  • This can be done on both Inputs and Outputs
  • Style guide recommends this should be avoided when possible

Example of export alias

@Output('myClick') clicks = new EventEmitter<string>()

Property Setters

  • Intercept input property changes can be done with a setter –
  • This is useful if you need to do some sort of validation or standardization on inputs
  • Angular Fundamentals class uses this to create a derived value from an input property
    • Uses this to set the color of a voting icon to set or not set
    • The setting on an input property toggles the color of the icon
    • Eliminates the need for a conditional on the icon
  • Parent components can bind to output event properties on children and react to those events

    • Example of an input setter. This can then be used to reference an internal variable _name

    @Input() set name(name: string) {...} get name(name: string) { ... }

    ContentChild Expand

    • @ContentChild
      • Returns a reference to the first matching component or directive within the content of your component
    • @ContentChildren
      • Returns a reference to all matching components or directives within the content of your component

    Parent / Child Communication Through Local Variables

    • Allows a parent component to interact with child properties and invoke child methods
      • This cannot be done with data binding since it cannot read child properties or invoke child methods
    • Add a local variable on the child component
      • This gives you a reference to the child component and the ability to access any of its properties or methods from within the parent template
    • All wiring must be done entirely within the parent template
      • The parent component itself has no access to the child
      • This cannot be used if an instance of the parent component class must read or write child component values or must call child component methods
    • To implement
      • Add a template reference variable to the child component
      • The parent template can now reference the public methods and properties on the child

    Example

    <countdown-timer #timer>

    Parent/Child Communication Through Service

    • Parents and children can communicate via a shared service whose interface enables bi-directional communication within the family
    • Useful to define a Subject within this service
      • Have components subscribe to an Observable from this subject
      • Use the next() method to send a message to all subscribed components
      • Make sure to remove all subscriptions in the ngOnDestroy hook

    ViewChild

    • @ViewChild and @ViewChildren are used to get a reference to components that you define in the view template of your component
    • Use this when the parent component class must refer to the child component class
    • Allows you to inject the child component into the parent as a ViewChild
      • Import references to the ViewChild decorator and the AfterViewInit lifecycle hook
      • Inject using the @ViewChild decorator
      • You can now reference the child component through this variable, but it is not available until after Angular displays the parent view
      • Need to use the ngAfterViewInit hook to work with it. The child component is not available until after Angular displays the parent view
      • This can also be done with templates by referencing a template reference

    Example

    @ViewChild(CountdownTimerComponent) private timerComponent: CountdownTimerComponent

    Styles

    • Use to isolate CSS rules to within the component template
      • Will not be inherited by any components nested within the template nor by any content projected into the components
      • By default, component CSS styles are encapsulated into the components view and don’t affect the rest of the application
    • Best practices
      • Do not override the styles of a component – consider then private implementation details for that component
      • If you are authoring a component that others will use, you can accept customization from consumers in a number of ways (see https://angular.io/guide/component-styles#authoring-a-component-to-support-customization)
    • Special selectors from the world of shadow DOM style scoping
      • :host
        • Host element – element that matches the components selector into which the template is rendered
        • This selector targets that host element
        • Rules will affect the host element and all its descendants
        • You can add selectors after the host selector to target elements within the component
      • :host(.active)
        • Function form of the host selector
        • Applies host styles conditionally
        • Includes another selector inside parentheses after :host
        • Example above targets the host element but only when it also has the active CSS class applied to it
      • :host-context
        • Use to apply styles based on some condition outside of a component’s view
        • :host-context(.theme-light) h2 would apply rules to all h2 elements inside the component only if some ancestor element has the CSS class theme-light
    • Ways to add styles for a component
      • By setting styles or styleUrls properties of the Component decorator
        • These are arrays or strings
        • If you are using the CLI to build, you can add sass, less, or stylus files into the styleUrls metadata with the appropriate extensions
        • Angular CLI build process runs the pertinent CSS preprocessor
      • Inline in the template HTML
        • Embed a style tag into the template or a link tag to reference an external stylesheet
    • With CSS imports
      • use to import CSS sheets into other CSS sheets
      • the URL is relative to the CSS file into which you are importing
    • To control how CSS encapsulation works on a component level, you can set the view encapsulation mode in the component metadata (encapsulation tag)
      • ViewEncapsulation.ShadowDom
        • Uses browser's native shadow DOM implementation. The component styles are included within the shadow DOM
        • Only works on browsers that have native support for shadow DOM, which is limited
      • ViewEncapsulation.Emulated
        • Default
        • Emulated the behavior of shadow DOM by preprocessing the CSS code to effective scope the CSS to the component's view
      • ViewEncapsulation.None
        • no view encapsulation. This behaves the same as pasting the component's styles into the HTML
    • If you need to apply styles across the entire page, add the rules to the site.CSS file

    Example

    @import './hero-details-box.css'"

    View Layer Architecture

    • Components can be separated by function
      • Keeps components small, simple and promotes reuse
    • Smart components
      • Aka application-level components, container components
      • Usually tied to an application since reuse is difficult
      • Has too many application-specific features, such as services injected into the constructor
      • Can be seen as a top-level component composed of one or more presentation components
        • Top level component of a route is usually a smart component
        • This component handles getting data from services
        • Data can be passed into a child presentation component for display
    • Presentation components
      • Aka Pure components, dumb components
      • Handles issues like formatting data into a table on the screen
      • Take data using inputs and emit events to
      • Try to make as application agnostic as possible
        • Avoid service injection that ties it to a specific application
        • Use inputs and outputs to get data into them
      • This is something that would be a good candidate for use in another application or in multiple places within the same application
    • One complication is event handling
      • Child components can only dispatch events to their parent component
      • Events do not bubble like they do in JS
      • A store-like service can be a solution if this is needed
    • These categories are guidelines or starting points
      • Smart components do not have to be top level component only
      • You may find yourself forced to make a presentation component into a smart component when need a service in it
      • It may not be worth trying to keep a component a presentation; you may sacrifice simple design for the sake of reusability
      • Design wins over reusability in this case
      • Start with a single top level smart component and split it into smaller presentation components as it gets bigger
      • With event handling, you could end up having to chain event listeners all the way down the component tree to stick with this paradigm
      • Probably better to make a leaf node a smart component in this case as it simplifies design
