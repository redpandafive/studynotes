Services

  • Basics
  • Configuration

Basics

  • Can be anything, but typically is a class with a narrow well-defined purpose
    • Could be any value, function, or feature that your application needs
  • These provide specific functionality not directly related to views
    • Components should limit its functionality to
      • Present properties and methods for data binding
      • Mediate between the view and application logic
  • Injected into components as dependencies to provide separation of logic
    • This is logic that can be shared across components
    • Promotes modularity, keeping components lean and efficient
    • Services can depend on other services, so you keep them focused and specialized
  • Services are singletons

Configuration

  • Identified by the @Injectable() decorator
    • Technically only required when you inject a service that has dependency services of its own
    • Best practice to just add it
  • To inject the service, list it as a parameter in the target class constructor
    constructor(private logger: Logger) { }
  • There are two ways to make a singleton service in Angular
    • Declare root for the value of the @Injectable providedIn property
    • Include the service in the AppModule or in a module that is only imported by the AppModule
  • Beginning with Angular 6, setting the providedIn to root is the preferred way since it makes them tree shakable
    • Pre-Angular 6, the preferred way is to register them in the providers array of modules
    • Setting providedIn to root makes the service available throughout the application
Simple example of a service class
import { Injectable } from '@angular/core';

@Injectable({
providedIn: 'root',
})
export class HeroService {
constructor() { }
}
