Context Object

  • Represents the session with the database, allowing querying and saving data
  • Defined by deriving from System.Data.Entity.DbContext
    • This exposes a typed DbSet<TEntity> for each class in our model.
  • This derived class controls how the base constructors
    • How the context will connect to a database â€” that is, how a connection string is found/used
    • Whether the context will use calculate a model using Code First or load a model created with the EF Designer
  • With no parameters called in the context constructor, EF will
    • Run in Code First mode
    • Use a database connection created by convention using either SQL Express or LocalDB
      • Visual Studio 2012 and later includes LocalDB - previous versions use SQL Express
      • If SQL Express is running, it will be used. If SQL Express is not available then LocalDB will be registered as the default instead
  • When a string is passed to the constructor containing the database name
    • EF will run in code first mode
    • A connection string to the specified name will be used to connect to SQL Express or LocalDB
  • Alternatively, a connection can be stored in a web.config or app.config file
    • This is an easy way to tell DbContext to use a database server other than SQL Express or LocalDB
    • If the name of the connection string matches the name of your context then it will be found by DbContext when the parameterless constructor is used.
    • If the connection string name is different from the name of your context then you can tell DbContext to use this connection in Code First mode by passing the connection string name to the DbContext constructor
